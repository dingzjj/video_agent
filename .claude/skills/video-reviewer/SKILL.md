# 视频自动审查与修复器

## 定位
**这不是打分工具，是自动修复工具。**

发现问题 → 直接修改对应的 `.tsx` 场景文件 → 保存 → 输出修改报告。
只有"无法自动修复"的问题才提建议给用户。

## 触发条件
- video-producer 完成后**自动调用**（无需用户手动触发）
- 用户说"检查一下视频" / "review 一下" / `/video-reviewer`

## 输入
读取以下文件：
- `output/[主题名]/script.json` — 口播内容基准（只读，不改）
- `output/[主题名]/remotion/src/scenes/*.tsx` — 场景代码（可修改）
- `output/[主题名]/remotion/src/Video.tsx` — 时间轴（可修改）

---

## 核心约束：口播保护原则

**口播（voiceover）和字幕（subtitle）内容绝对不能修改。**

原因：口播未来会并行调用 TTS 配音服务，视觉改动必须配合口播时长，而不是反过来。

具体规则：
- ✅ 可以改：视觉样式、字号、颜色、动效参数、布局结构、背景
- ✅ 可以改：场景时长（`durationInFrames`）±10% 微调以改善节奏
- ❌ 不能改：`voiceover` 字段内容
- ❌ 不能改：`subtitle` 字段内容
- ❌ 不能改：场景数量和顺序（会影响口播对应关系）
- ❌ 不能改：场景时长超过 ±20%（会导致口播与画面不同步）

---

## 检查维度与自动修复规则

### 维度 1：AI 味检查（最高优先级）

**检查项**：
- [ ] 同一个场景是否用了三个相同结构的卡片（图标+标题+描述并排）
- [ ] 连续两个以上场景背景颜色完全相同
- [ ] 卡片背景透明度低于 0.1（`rgba(color, 0.06)` 这类）
- [ ] 主标题字号小于 64px

**自动修复**：
- 三卡片并排 → 改为逐句大字型（参考 `component-templates.md` 模板2）
- 背景相同 → 偶数场景改为主题深色彩背景
- 低透明度卡片 → 要么提高到 0.15+，要么去掉背景色直接裸排
- 字号太小 → 主标题提升到 80px+

---

### 维度 2：信息密度检查

**检查项**：
- [ ] 单个场景内文字行数是否超过 6 行
- [ ] 单行字符数是否超过 18 字（竖屏标准）
- [ ] 一个场景是否同时有：大标题 + 副标题 + 多卡片 + 底部说明（信息超载）

**自动修复**：
- 超过 6 行 → 删除辅助说明文字，只保留核心 1-2 句
- 单行超 18 字 → 拆行或缩短
- 信息超载 → 删除底部说明文字，保留主卡片内容

---

### 维度 3：动效质量检查

**检查项**：
- [ ] 是否使用了 CSS transition 或 Tailwind `animate-*` 类（Remotion 不支持）
- [ ] spring 参数是否用了合理预设（避免 `damping: 100` 这个非标准值）
- [ ] 所有动画是否都基于 `useCurrentFrame()`
- [ ] 是否有 `premountFor={30}` 硬编码（应该用 `1 * fps`）

**自动修复**：
- CSS transition → 删除，改用 spring/interpolate
- `damping: 100` → 改为 `damping: 200`（smooth 预设）或 `damping: 20, stiffness: 200`（snappy 预设）
- `premountFor={30}` → 改为 `premountFor={1 * fps}`

---

### 维度 4：布局节奏检查

**检查项**：
- [ ] 是否连续 3 个以上场景用了相同布局模板
- [ ] Hook 场景（第一个）是否有足够大的主标题（≥80px）
- [ ] Payoff 场景（最后一个）是否足够极简（不超过 2 个文字元素）

**自动修复**：
- 连续相同布局 → 中间插入大字场景替换其中一个卡片场景
- Hook 字号不足 → 提升至 88-96px
- Payoff 太复杂 → 删除多余元素，只保留核心一句话

---

### 维度 5：视觉一致性检查

**检查项**：
- [ ] 是否所有场景都加载了字体（`loadFont()`，不是裸字符串）
- [ ] 强调色在不同场景间是否一致
- [ ] 光晕装饰是否每个场景都有（至少 1 个）

**自动修复**：
- 缺少 `loadFont()` → 在文件顶部添加正确的 google-fonts 加载
- 缺少光晕 → 在 AbsoluteFill 内添加标准光晕 div

## 工作流程

### Step 1: 读取所有场景文件
```
读取 script.json 提取口播内容作为基准（只读）
读取 src/scenes/*.tsx 所有场景代码
读取 src/Video.tsx 时间轴结构
```

### Step 2: 逐项检查，标记问题
按 5 个维度逐场景扫描，记录：
- 场景 ID
- 问题类型
- 是否可自动修复
- 修复方案

### Step 3: 自动修复（直接编辑 .tsx 文件）
按优先级顺序修复：
1. AI 味问题（最影响观感）
2. 动效规范问题（影响渲染正确性）
3. 信息密度问题
4. 布局节奏问题
5. 视觉一致性问题

**修复时必须检查**：本次修改是否影响了场景时长？如有影响，同步更新 `Video.tsx` 中的 `durationInFrames`，但不超过原值 ±20%。

### Step 4: 二次验证
修复完后重新扫描一遍，确认问题已解决，没有引入新问题。

### Step 5: 输出报告
生成 `output/[主题名]/review.md`

## 常见问题

**Q: 修复布局时怎么保证口播节奏不乱？**
A: 只改视觉代码，不改时长。如果信息删减导致画面"太空"，用更大的字号和更慢的动效填满时间，而不是删除场景。

**Q: 三卡片改成大字，内容放不下怎么办？**
A: 每张卡片的内容取最核心的一句（通常是标题），副标题/描述文字直接删除。信息量收敛是正确方向，不是问题。

**Q: 发现脚本本身有问题（比如逻辑不清）怎么办？**
A: 在"人工处理项"里说明，建议用户重新运行 `/video-script-generator`，不要私自改动口播内容。

**Q: 如果某个场景的问题改了会影响其他场景怎么办？**
A: 先修当前场景，修完后检查前后场景的一致性（颜色、字体、强调色），确保视觉连贯。
